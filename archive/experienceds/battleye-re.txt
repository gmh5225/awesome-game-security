## Highlights

- 129 kernel APIs resolved via `MmGetSystemRoutineAddress` + `FltGetRoutineAddress`
- 13 IOCTL handlers on `\Device\BattlEye`, all METHOD_NEITHER
- Custom BattlEye VM in `.be0` (not VMProtect, not Themida) — 42K pushfq, 4.6 MB monster function
- Zero BEDaisy-installed hooks in `HalDispatchTable` / `HalPrivateDispatchTable`
- Security cookie deterministic: `RDTSC XOR (base + 0x1C120)`, 48-bit entropy
- Hardcoded target string `enlisted.exe` at `.data + 0x1C1A0`, 5 XREFs

## Important observations

**PatchGuard coverage:** protects `.text`, `.INIT`, imports/exports, and various ntoskrnl/HAL globals — but does **NOT** protect drivers' own `.data`/`.rdata` sections. This is why DMA writes to `.data` (like patching the enlisted.exe string) are safe from PatchGuard BSODs.

**Big .be0:** the massive VM section makes standard PE tools fail (revula/pefile/LIEF all designed for typical drivers). Memory-mapped format also complicates parsing — sections at VA, not raw file offsets.

**Two exception directories:** the `.pdata` section is a small subset (`.text` prologs), the real one is at RVA `0x786180` (in .be0's data area) with 226 valid function entries covering the whole driver.

# API Resolution — 3-tier scheme

BEDaisy resolves ~135 kernel APIs from just 2 statically imported functions.

## Tier 1 — Static IAT (6 functions)

Located at `.rdata` `0x1A000`, 64 bytes total.

```
IAT[0] @ 0x1A000: FLTMGR.SYS!FltGetRoutineAddress
IAT[1] @ 0x1A008: ntoskrnl.exe!MmGetSystemRoutineAddress
IAT[2] @ 0x1A010: ntoskrnl.exe!__C_specific_handler
IAT[3] @ 0x1A018: ntoskrnl.exe!__chkstk
IAT[4] @ 0x1A020: ntoskrnl.exe!MmMapLockedPagesSpecifyCache
IAT[5] @ 0x1A028: ntoskrnl.exe!KeBugCheckEx
```

Only `FltGetRoutineAddress` and `MmGetSystemRoutineAddress` are used to resolve everything else. The other 4 are helpers (SEH handler, stack probe, cache mapping, forced BSOD for tamper).

**Attack surface:** patching IAT[0] or IAT[1] to a NULL/stub function completely blinds BEDaisy — no more dynamic API resolution.

## Tier 2 — Plaintext 16-char blocks (@ 0x1A050)

Names stored truncated to 16 bytes each (contiguous in memory):

```
0x1A050: FltGetRequestorP    → FltGetRequestorProcess(Id)
0x1A060: FltReleaseFileNa    → FltReleaseFileNameInformation
0x1A070: FltRegisterFilte    → FltRegisterFilter
0x1A080: FltUnregisterFil    → FltUnregisterFilter
0x1A090: NtReadVirtualMem    → NtReadVirtualMemory
0x1A0A0: FltGetFileNameIn    → FltGetFileNameInformation
0x1A0B0: FltStartFilterin    → FltStartFiltering
0x1A0C0: FltQueryInformat    → FltQueryInformationFile
```

Then more (`.rdata + 0x17F`):
```
0x1A17F: ZwSuspendThread\0
0x1A190: NtWriteVirtualMe    → NtWriteVirtualMemory
0x1A1A0: ZwGetContextThre    → ZwGetContextThread
0x1A1B0: ZwProtectVirtual    → ZwProtectVirtualMemory
0x1A1C0: NtDeviceIoContro    → NtDeviceIoControlFile
```

Each 16-char block is passed to `MmGetSystemRoutineAddress` or `FltGetRoutineAddress`. The truncation is deliberate (obfuscation-lite) — resolution succeeds because the OS API takes a `UNICODE_STRING*` and reads the length prefix, so trailing bytes don't matter.

## Tier 3 — NOT-encoded (0xFF XOR, @ 0x1A0D0)

Simple encoding: `decoded_byte = 0xFF - encoded_byte`.

Raw encoded bytes:
```
0x1A0D0: b7 9e 93 af 8d 96 89 9e 8b 9a bb 96 8c 8f 9e 8b
0x1A0E0: b7 9e 93 bb 96 8c 8f 9e 8b 9c 97 ab 9e 9d 93 9a

0x1A100: b4 a8 b6 b1 cc cd b4 ba b1 b8 a0 b6 b1 ab ba ad
0x1A110: c0 98 bb 87 98 94 a8 96 91 cc cd 94 ba 91 98 b6
0x1A120: bf cc aa a0 bb a7 b8 b4 a8 b6 b1 cc cd b4 a0 b6
0x1A130: 91 8b 9a 8d 99 9e 9c 9a bf bf cc aa a0 bb a7 b8
0x1A140: c0 98 bb 87 98 94 b6 91 8b 9a 8d 99 9e 9c 9a bf
```

Decoded:
```
0x1A0D0: HalPrivateDispat   → HalPrivateDispatchTable
0x1A0E0: HalDispatchTable
0x1A0F0: DxgCoreInterface
0x1A100: KWIN32KENG_INTER   → KWIN32KENG_INTERFACE

These are MSVC-mangled globals (`?g` = variable, `@@3U..@@A` = struct type):

| Symbol | Kind | Used For |
|--------|------|----------|
| `HalPrivateDispatchTable` | ntoskrnl export | **READ** for integrity check |

| `?gDxgkInterface@@3U_DXG*` | dxgkrnl global | Dxgk interface |

# IOCTL Dispatch Table

Device: `\Device\BattlEye` (Unicode string @ `.data` `0x1C230`)
Symbolic link: `\DosDevices\BattlEye` → usermode `\\.\BattlEye`

## IOCTL code decoding

```c
#define CTL_CODE(DeviceType, Function, Method, Access) \

- **DeviceType** = `0x22` (FILE_DEVICE_UNKNOWN)

## METHOD_NEITHER buffer semantics

```c
IRP->AssociatedIrp.SystemBuffer                       = NULL
IrpSp->Parameters.DeviceIoControl.Type3InputBuffer    = <user pointer, INPUT>
IRP->UserBuffer                                       = <user pointer, OUTPUT>
```

Driver must perform `ProbeForRead` / `ProbeForWrite` on both pointers before dereferencing. Missing probe on any handler = arbitrary kernel R/W primitive.

## Comparison site byte context

IOCTL codes appear as 32-bit immediates in `.be0`. Example — `0x0022001F` @ `0x311AAD`:

```
Prev tail:  98 e6 cc 8d 48 23 f3 e9
IOCTL bytes: 1f 00 22 00                     (0x0022001F, little-endian)
Handler:     41 bb 24 38 8e e7 44 8b 14 27   (mov r11d, 0xe78e3824; mov r10d, [rdi+riz])
```

Compare patterns to search for in `.be0`:
- `81 F8 XX XX XX XX` — CMP EAX, imm32
- `81 F9 XX XX XX XX` — CMP ECX, imm32
- `3D XX XX XX XX` — CMP EAX, imm32 (short form for EAX)

## Dispatch structure

`DriverObject->MajorFunction[IRP_MJ_DEVICE_CONTROL]` = handler inside `.be0` (obfuscated, exact dispatcher VA not statically identified).

Handlers call `IofCompleteRequest` (resolved API pointer at `.data + 0x1C380`) — this is a XREF anchor for finding the dispatcher.

## Access

`CreateFileW(L"\\\\.\\BattlEye", ...)` may fail with `ERROR_ACCESS_DENIED` if `IoCreateDevice` sets a restrictive SECURITY_DESCRIPTOR. The intended client is BEService.exe.

## VM data decoded as instructions

`.be0` is a mix of real code and VM bytecode / immediate data. The x86 decoder, scanning the byte stream linearly, finds "instructions" inside VM data blobs.

Example around `.be0 0x29A96` (labeled as WBINVD in the raw byte scan):

```
Before:  ... 02 fb d7 55 1a 22 02 fb (repeating 0x2202fb pattern)
Match:   0f 09 6a 22 02 fb 6c e1 ba 22 02 fb 27 09 02 22
```

The `0x2202fb` sequence repeats dozens of times in the surrounding bytes — this is VM bytecode / immediates, not executable code. The x86 decoder reads it as `0F 09` (WBINVD) inside data.

## Real instruction distribution

Filtered by surrounding valid x86 context:

- `.text`: 5 real INVD, ~2 real WBINVD
- `INIT`: 0 anti-DMA instructions
- `.be0`: ~5-10 real INVD (rough estimate), majority are decoder false positives from VM data

## INVD/WBINVD vs FPGA DMA

- INVD/WBINVD flush CPU L1/L2/L3 caches
- FPGA DMA reads physical memory over PCIe, bypassing CPU cache hierarchy
- DMA controller communicates with memory controller directly (chipset level)
- Cache state is irrelevant for DMA reads

DMA writes are cache-coherent by chipset (cache snoop protocol). INVD would break this coherency and cause data loss if used routinely during runtime — not viable as a defensive technique.

## Chipset-level anti-DMA

- VT-d / AMD-Vi (IOMMU) — remaps PCIe device memory access via translation tables
- Windows Kernel DMA Protection (Windows 10 1803+) — enforces IOMMU for hot-plug PCIe devices at runtime
- BIOS/UEFI configuration — disables DMA from external ports, enables IOMMU by default

Driver-level INVD/WBINVD does not replicate any of these.

# Security Cookie

MSVC `/GS` stack protection cookie, kernel variant.

## Location

```
Cookie          @ .data + 0x1C120
Complement      @ .data + 0x1C128
```

Invariant: `cookie XOR complement == 0xFFFFFFFFFFFFFFFF`.

## Session 1 values (base `0xFFFFF80A68E80000`)

```
Cookie:      0x00004D776ABDD308
Complement:  0xFFFFB28895422CF7
XOR check:   0xFFFFFFFFFFFFFFFF  OK
```

## Derivation formula

```
cookie       = RDTSC_at_init XOR (image_base + 0x1C120)
cookie[63:48] = 0    (top 16 bits masked)
```

`__security_init_cookie` code at `INIT + 0x005` (after the initial `JMP` to `.be0`):

```asm
INIT:0x1F005:
push   rax
...
mov    rax, [rip + off_cookie]     ; load current cookie
xor    r9d, r9d
movabs r8, 0x002B992DDF A232       ; default cookie value
test   rax, rax
jz     +5
cmp    rax, r8                     ; already customized?
jnz    +0x38                       ; if yes, skip re-init

rdtsc                              ; EDX:EAX = 64-bit TSC
shl    rdx, 32
lea    rcx, [rip + cookie_var]     ; RCX = kernel VA of cookie
or     rax, rdx                    ; RAX = TSC combined
xor    rax, rcx                    ; RAX = TSC XOR (kernel VA)
mov    [rip + cookie_var], rax
mov    word ptr [rip + cookie_top_2], r9w  ; zero top 16 bits
```

## Session 1 derived RDTSC

```
RDTSC_at_init = cookie XOR (image_base + 0x1C120)
              = 0x00004D776ABDD308 XOR 0xFFFFF80A68E9C120
              = 0xFFFFB57D02541228
```

## Entropy

- Top 16 bits = 0 (masked)
- Lower 48 bits = RDTSC XORed with known constant (kernel VA post-ASLR)
- Practical effective entropy ≈ 32-36 bits given RDTSC jitter and rough boot-time estimates

## LoadConfig references

Load Config Directory (`RVA 0x1A210`, size 264 bytes):

```
SecurityCookie                  = 0xFFFFF80A68E9C120  (session 1 kernel VA)
GuardCFCheckFunctionPointer     = 0xFFFFF80A68E9A040
GuardCFDispatchFunctionPointer  = 0xFFFFF80A68E9A048
GuardFlags                      = 0x100  [CF_INSTRUMENTED]
```

Pointers at `.rdata` `0x1A040` and `0x1A048` hold Control Flow Guard check/dispatch function pointers, populated by the OS loader.

Six pool-allocated handles in `.data`:

- Calling `ObUnRegisterCallbacks(handle)` from kernel context
- Patching the callback's function pointer in its registration structure
- Walking the kernel's internal notify arrays and clearing entries
