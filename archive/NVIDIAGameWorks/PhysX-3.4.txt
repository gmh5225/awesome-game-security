PhysX is a physics simulation engine developed by NVIDIA that provides realistic physics simulation capabilities for games and other applications. This document provides a high-level overview of the PhysX SDK 3.4 architecture, core components, and supported platforms. For detailed information about specific subsystems, please refer to their respective wiki pages.

PhysX enables developers to simulate rigid body dynamics, continuous collision detection, scene queries, vehicles, cloth, and particles. This overview focuses on the key architectural components and their relationships, serving as a starting point for understanding the SDK.

## Architecture Overview

PhysX SDK follows a component-based architecture with several core systems working together to provide physics simulation capabilities.

### Collision Detection System

PhysX uses a multi-staged collision detection system to efficiently detect and resolve collisions between objects.

```mermaid

        MidPhase["Mid Phase\n(Shape-specific culling)"] 
        NarrowPhase["Narrow Phase\n(PCM/Contact Generation)"]
        CCD["Continuous Collision\nDetection (CCD)"]
        
        Input --> BroadPhase
        BroadPhase --> MidPhase
        MidPhase --> NarrowPhase
        NarrowPhase --> Output
        NarrowPhase --> CCD
        CCD --> Output

    end
    
    Output --> Solver["Constraint Solver"]
```

Sources: [PhysX_3.4/Source/GeomUtils/src/ccd/GuCCDSweepConvexMesh.cpp:58-629]()

The collision detection pipeline consists of:

1. **Broad Phase**: Quickly identifies potential collision pairs using spatial partitioning techniques like Multi-Broad Phase (MBP) or Sweep-and-Prune (SAP)
2. **Mid Phase**: Shape-specific culling for complex geometries like meshes and heightfields
3. **Narrow Phase**: Detailed collision analysis using algorithms like PCM (Persistent Contact Manifold)
4. **Continuous Collision Detection (CCD)**: Prevents tunneling for fast-moving objects

## Initialization Example

To initialize PhysX and create a basic simulation, developers typically follow this pattern:

```
// Create foundation and physics objects
PxFoundation* foundation = PxCreateFoundation(PX_PHYSICS_VERSION, allocator, errorCallback);
PxPhysics* physics = PxCreatePhysics(PX_PHYSICS_VERSION, *foundation, PxTolerancesScale(), trackOutstandingAllocations, pvd);

// Create scene
PxSceneDesc sceneDesc(physics->getTolerancesScale());
sceneDesc.gravity = PxVec3(0.0f, -9.81f, 0.0f);
sceneDesc.cpuDispatcher = PxDefaultCpuDispatcherCreate(numThreads);
sceneDesc.filterShader = PxDefaultSimulationFilterShader;
PxScene* scene = physics->createScene(sceneDesc);

// Create actors and add to scene
PxRigidDynamic* body = physics->createRigidDynamic(PxTransform(position));
PxShape* shape = physics->createShape(PxSphereGeometry(radius), *material);
body->attachShape(*shape);
body->setMass(mass);
scene->addActor(*body);

// Simulation loop
scene->simulate(timestep);
scene->fetchResults(true);
```

### Core Systems Organization

PhysX is organized into several core systems that work together to perform physics simulation:

1. **Collision Detection System** - A multi-phase pipeline for detecting collisions between objects
2. **Rigid Body Dynamics** - Handles physical simulation of rigid bodies
3. **Scene Query System** - Provides spatial queries like raycasts, sweeps, and overlaps
4. **Simulation Controller** - Orchestrates the entire simulation process
5. **Extensions** - Optional components that add higher-level functionality
6. **Debugging Tools** - Visualization and profiling tools

Each of these systems employs its own set of design patterns and algorithms to efficiently solve specific physics problems.

### PxShared Foundation

Starting with PhysX 3.4, common functionality was moved to a shared library called "PxShared". This includes:

- Core types and utilities
- Memory management
- Task system
- Foundation classes
- Math utilities

This architectural change allows multiple components (like PhysX and APEX) to use the same foundation code.

Sources: [PhysX_3.4/release_notes.html:835-839]()

### Factory Pattern

The Factory pattern is used extensively to create PhysX objects. The `PxPhysics` class serves as the primary factory for creating scenes, actors, and other objects:

```
PxPhysics* physics = PxCreatePhysics(...);
PxScene* scene = physics->createScene(...);
PxRigidDynamic* actor = physics->createRigidDynamic(...);
PxMaterial* material = physics->createMaterial(...);
```

This pattern centralizes object creation and ensures proper initialization and tracking of resources.

### Observer Pattern

PhysX uses the Observer pattern for event notifications. Applications can register callback interfaces to receive notifications about physics events:

- `PxSimulationEventCallback` - For contact, trigger, constraint events
- `PxContactModifyCallback` - For modifying contacts before resolution
- `PxCCDContactModifyCallback` - For modifying CCD contacts

This allows applications to react to physics events without polling.

### Strategy Pattern

The Strategy pattern is employed to select different algorithms for the same purpose:

- `PxBroadPhaseType` - Different broad phase collision detection algorithms
- `PxPruningStructureType` - Different pruning structures for scene queries
- `PxMeshMidPhase` - Different midphase structures for triangle meshes

This allows PhysX to adapt to different performance requirements and hardware capabilities.

Sources: [PhysX_3.4/release_notes.html:722-835]()

### Reference Counting

Shared resources like triangle meshes, convex meshes, materials, and heightfields use reference counting to manage their lifecycle. This allows multiple objects to share these resources without duplication:

```
// Both shapes share the same material
PxMaterial* material = physics->createMaterial(0.5f, 0.5f, 0.1f);
PxShape* shape1 = physics->createShape(geometry1, *material);
PxShape* shape2 = physics->createShape(geometry2, *material);
```

PhysX 3.4 introduced `acquireReference()` to explicitly increment reference counts when needed.

Sources: [PhysX_3.4/release_notes.html:729-732]()

## Task-Based Concurrency

PhysX uses a task-based concurrency model to parallelize physics computation:

```mermaid
graph TD
    Simulate["PxScene::simulate()"] --> TaskManager["TaskManager"]
    TaskManager --> DispatchTasks["Dispatch Tasks"]
    
    DispatchTasks --> BP["Broad Phase Task"]
    DispatchTasks --> NP["Narrow Phase Tasks"]
    DispatchTasks --> Solve["Solver Tasks"]
    DispatchTasks --> Integrate["Integration Tasks"]
    
    BP --> |"Dependencies"| NP
    NP --> |"Dependencies"| Solve
    Solve --> |"Dependencies"| Integrate
    
    Integrate --> Complete["Completion Task"]
    Complete --> FetchResults["PxScene::fetchResults()"]
```

This approach allows PhysX to efficiently utilize multiple CPU cores while maintaining proper dependencies between different stages of the simulation.

PhysX 3.4 introduced an alternative simulation API with `collide()`, `fetchCollision()`, and `advance()` functions to give applications more control over the simulation pipeline.

### Continuous Collision Detection (CCD)

CCD prevents fast-moving objects from tunneling through thin objects. PhysX 3.4 implements CCD through sweep tests:

```mermaid
sequenceDiagram
    participant Shape1 as "Moving Shape"
    participant Mesh as "Triangle Mesh"
    participant CCD as "SweepAnyShapeMesh"
    participant Filter as "Spatial Filtering"
    participant Test as "Contact Generation"
    
    Shape1->>CCD: Request CCD sweep
    CCD->>Filter: Compute swept bounds
    Filter->>Mesh: Find potential triangles
    Mesh-->>Filter: Return triangle indices
    
    loop For each potential triangle
        Filter->>Test: Test triangle against shape
        Test-->>CCD: Return time of impact
    end
    
    CCD-->>Shape1: Return earliest contact
```

The key functions implementing CCD sweeps are:
- `SweepAnyShapeMesh` - Performs detailed sweep tests against triangle meshes
- `SweepEstimateAnyShapeMesh` - Performs fast, conservative estimates for early rejection
- `SweepShapeTriangle` - Tests a specific shape against a single triangle

Sources: [PhysX_3.4/Source/GeomUtils/src/ccd/GuCCDSweepConvexMesh.cpp:438-630]()

### CCD Workflow

The `SweepAnyShapeMesh` function demonstrates the workflow for CCD collision detection with triangle meshes:

1. **Compute Swept Bounds**: Calculate the bounding box that encompasses the shape's entire motion path
2. **Transform to Mesh Space**: Convert the swept bounds to the mesh's local space
3. **Triangle Filtering**: Use the mid-phase to identify potentially colliding triangles
4. **Fast AABB Tests**: Perform quick AABB-vs-AABB tests to eliminate non-colliding triangles
5. **Sort Triangles**: Order triangles by estimated time of impact
6. **Exact Sweep Tests**: Perform precise sweep tests against individual triangles
7. **Find Earliest Hit**: Return the earliest collision time, point, normal, and face index

Sources: [PhysX_3.4/Source/GeomUtils/src/ccd/GuCCDSweepConvexMesh.cpp:438-630]()

### Optimization Techniques

The CCD implementation employs several optimization techniques:

1. **Two-phase Testing**: Initial coarse AABB tests followed by exact GJK-based tests
2. **Bounding Volume Hierarchies**: Efficient triangle filtering with BVH structures
3. **Triangle Sorting**: Process triangles in order of estimated time of impact
4. **In-sphere Tests**: Special handling for initially overlapping cases
5. **Early Exit**: Stop processing once a definitive hit is found

### Object Lifecycle

Objects go through several operations in the MBP system:

1. **Addition**: When objects are added to the system, the MBP:
   - Creates an `MBP_Object` instance
   - Identifies regions the object overlaps with
   - Adds the object to each overlapping region
   - Assigns handles to track region membership

2. **Update**: When objects move, the MBP:
   - Updates the object's AABB
   - Re-evaluates which regions the object overlaps
   - Adds to newly overlapping regions
   - Removes from no longer overlapping regions
   - Updates the object's position within each region

3. **Removal**: When objects are removed:
   - The object is removed from all regions it belongs to
   - Its MBP_Object is marked as removed
   - Its memory is recycled

### Box Pruning Algorithms

The MBP uses efficient box pruning algorithms to minimize the number of AABB tests:

1. **Complete Box Pruning**: For dynamic-dynamic overlap tests
   - Objects are sorted along the x-axis
   - Only tests objects with overlapping x-extents
   - Uses "sweep and prune" concept

2. **Bipartite Box Pruning**: For dynamic-static overlap tests
   - Separately handles dynamic and static objects
   - Exploits the fact that static objects don't move

These algorithms reduce the complexity from O(nÂ²) to approximately O(n log n) plus the number of actual overlaps.

## Purpose and Scope

This document describes the Continuous Collision Detection (CCD) system within PhysX 3.4. The CCD system prevents fast-moving objects from tunneling through each other or through thin objects, a problem that can occur with discrete collision detection alone. While discrete collision detection only checks for collisions at fixed time steps, CCD performs collision checks along an object's entire movement path.

This document focuses specifically on the sweep-based CCD implementation in PhysX 3.4. For information about the general collision detection pipeline, see [Collision Detection System](#2).

## Overview

In traditional physics simulations using discrete time steps, objects move instantaneously from one position to the next. If an object moves fast enough, it can completely pass through another object between time steps, resulting in a "tunneling" effect where no collision is detected.

The PhysX CCD system addresses this issue by performing swept collision tests along an object's movement path, ensuring that all potential collisions are detected regardless of an object's speed.

```mermaid

    E --> F["Compute Time of Impact (TOI)"]

    G --> H["Apply CCD Response"]
    H --> I["End CCD Process"]

```

### 1. Preparation Phase

The algorithm starts by computing the relative movement between the shapes:

```
PxVec3 relTr = trA - trB;
```

It then constructs a swept bounding box that encompasses the moving object along its entire path.

### 2. Broad Phase

The algorithm transforms the swept bounding box into the appropriate space (vertex space for meshes) and inflates it by the contact distance. It then performs a broad phase query to identify all triangles that potentially intersect with the swept box.


The algorithm sorts the potential collision triangles by their approximate time of impact (TOI) to optimize the narrow phase processing:

```
if(toi <= 1.f)
```

## Enabling CCD in PhysX

There are two primary ways to enable CCD in PhysX:

1. **Scene-level CCD**: Enable CCD for the entire scene using `PxSceneFlag::eENABLE_CCD`
2. **Object-level CCD**: Enable CCD for specific rigid bodies using `PxRigidBodyFlag::eENABLE_CCD`

Additionally, PhysX 3.4 introduced a new speculative CCD approach:

```
PxRigidBodyFlag::eENABLE_SPECULATIVE_CCD
```

This is a slightly cheaper, less robust solution that functions as an extension to the discrete time-stepping scheme and doesn't require enabling CCD at the scene level.

The CCD system generates contacts at the time of impact. These contacts include:

## Performance Considerations

CCD significantly increases simulation accuracy for fast-moving objects but comes with additional computational cost. To optimize performance:

1. Only enable CCD for objects that need it (fast-moving objects or objects with complex interactions)
2. Consider using `eENABLE_SPECULATIVE_CCD` for less critical scenarios
3. Adjust contact distance and solver parameters to balance accuracy and performance

## Limitations and Edge Cases

While CCD significantly improves collision detection for fast-moving objects, it has some limitations:

1. Increased computational cost, especially with many fast-moving objects
2. May not perfectly handle multi-object collision chains in a single frame
3. Can sometimes produce different behavior compared to discrete collision detection

PhysX 3.4 introduced the `PxRigidBodyFlag::eENABLE_CCD_MAX_CONTACT_IMPULSE` flag to enable the `maxContactImpulse` member of `PxRigidBody` to be used in CCD. This can be useful in scenarios like shooting a small ball through a plate glass window but may result in behavioral artifacts.

### Node Structure

The AABB tree is composed of two types of nodes:

### Insertion

The `insert()` method adds a new object to the tree:

1. If the tree is empty, create a root leaf node containing the object
2. If the root is a leaf with space available, add the object to the root
3. If the root is a leaf with no space, split the leaf and redistribute objects
4. If the root is an internal node, traverse the tree to find the best leaf for insertion:
   - If the chosen leaf has space, add the object
   - If not, split the leaf and redistribute objects

When inserting, the tree always attempts to place objects near similar objects to maintain locality and optimize spatial queries.

Sources: [PhysX_3.4/Source/SceneQuery/src/SqIncrementalAABBTree.cpp:538-667]()

### Updating Objects

The tree provides two methods for updating objects that have moved:

1. `update()` - Full update that removes and reinserts the object
2. `updateFast()` - Optimized update that keeps the object in its current leaf if its new bounds still fit

The update flow looks like this:

```mermaid
flowchart TD
    Start([Start Update]) --> CheckFast{"Fast update?"}
    
    CheckFast -->|Yes| BoundsCheck{"New bounds fit\nin current node?"}
    BoundsCheck -->|Yes| UpdateBounds["Update node bounds"]
    BoundsCheck -->|No| FullUpdate["Remove & re-insert object"]
    
    CheckFast -->|No| FullUpdate
    
    UpdateBounds --> PropagateChanges["Propagate changes up the tree"]
    FullUpdate --> End([End Update])
    PropagateChanges --> End
```

The fast update can significantly improve performance for objects that move small distances, as it avoids the cost of removal and reinsertion.

### Removal

The `remove()` method removes an object from the tree:

1. If the leaf contains multiple objects, simply remove the target object
2. If it's the last object:
   - If it's the root, release the node and set root to NULL
   - Otherwise, replace the parent with its other child (promoting the sibling)

After removal, the tree updates the AABB bounds up the hierarchy to ensure they remain tight.

## Performance Considerations

The Incremental AABB Tree makes several design choices to optimize performance:

1. **Leaf capacity**: Each leaf can hold up to 4 primitives, reducing tree depth and traversal costs
2. **Memory pools**: Custom pools reduce allocation overhead for nodes and indices
3. **Traversal optimizations**: Methods like `nodeIntersection()` and `traversalDirection()` use SIMD operations for faster intersection tests
4. **Incremental updates**: Objects that move small distances can use `updateFast()` to avoid expensive tree restructuring
5. **Split heuristics**: The `splitLeafNode()` method uses the axis of greatest extent for splitting, which tends to create more balanced trees

For large dynamic scenes, these optimizations are crucial to maintain interactive frame rates.

## Performance Considerations

The Contact System is designed with performance in mind:

1. **Lazy Evaluation**: Contact data is only processed when explicitly requested through the iterator
2. **Batched Processing**: Contacts are processed in pairs to improve cache coherency
3. **Memory Reuse**: The system reuses memory for contact objects rather than creating new instances
4. **Streaming Access**: The underlying contact data is accessed through a streaming iterator to minimize memory usage

These optimizations allow the system to efficiently handle simulations with thousands of contact points while providing a clean, easy-to-use interface.

## Field Samplers

Field Samplers create force fields and other influences that affect particle behavior after emission. They are crucial for creating complex, physically plausible effects by providing directional forces, turbulence, attraction, and other phenomena.

Major field sampler types:

### AttractorFieldSampler
Creates forces that attract or repel particles from a point or area. Configured with `AttractorFSAssetParams`.

### ForceField
Applies directional or radial forces to particles. Includes parameters like:
- `RadialForceFieldKernelParams`: Defines how force radiates
- `ForceFieldFalloffParams`: Controls how force diminishes with distance
- `ForceFieldNoiseParams`: Adds variation to the force

### HeatSource
Simulates thermal effects that cause particles to rise. Defined with `HeatSourceAssetParams` and can use geometries like `HeatSourceGeomSphereParams`.

### TurbulenceFieldSampler
Adds chaotic, swirling motion to particles. Configured with `TurbulenceFSAssetParams` and supports different resolutions (EGR_16, EGR_64, EGR_128).

### 4.3 Common Field Sampler Types in Detail

#### 4.3.1 Attractor Field Sampler

The Attractor Field Sampler (`AttractorFSAssetParams`) pulls particles toward one or more points in space, creating effects like particles being sucked into a vacuum or gravitating toward objects.

Key parameters:
- Strength: Force magnitude
- Falloff: How force diminishes with distance
- Position: Location of the attractor point

#### 4.3.2 Force Field

Force Fields (`ForceFieldAssetParams`) apply directional forces to particles, pushing them in specific directions. They can use various kernels and falloff patterns:

- `RadialForceFieldKernelParams`: Forces emanate from a central point
- `ForceFieldFalloffParams`: Controls how force diminishes with distance ("linear" falloff is common)
- `ForceFieldNoiseParams`: Adds noise to the force field for less uniform effects

#### 4.3.3 Turbulence Field Sampler

Turbulence Field Samplers (`TurbulenceFSAssetParams`) create chaotic, swirling movements in particles. They're commonly used for smoke, fire, and other fluid-like effects.

The database includes specialized turbulence configurations:
- `defaultTurbulenceFieldSampler`: Basic turbulence
- `DensitySmoke`: Optimized for smoke effects
- `pulseblastTurbulence`: For explosion effects

#### 4.3.4 Heat Source

Heat Sources (`HeatSourceAssetParams`) simulate thermal effects, causing particles to rise and disperse like heat from a fire or hot surface.

The database includes several heat source configurations:
- `defaultHeatSource`: Basic heat effect
- `DensitySmokeHeatSource`: For smoke simulations
- `hotlavaheatsource`: For lava effects
- `pulseblastheatsource`: For explosion effects
- `smokeHeatSource`: Specialized for smoke

Mesh rendering represents particles using 3D models instead of flat sprites. This is used for debris, fragments, and other complex particles:

- Each particle instance uses a 3D mesh
- Realistic behavior for solid objects like debris
- Supports material mapping with textures

### Using the PhysX Visual Debugger

To use PVD in your application:

1. **Initialize the connection**:
   ```cpp
   // Create PVD
   PxPvd* pvd = PxCreatePvd(foundation);
   
   // Setup connection transport (typically TCP)
   PxPvdTransport* transport = PxDefaultPvdSocketTransportCreate("localhost", 5425, 10);
   
   // Connect to PVD
   pvd->connect(*transport, PxPvdInstrumentationFlag::eALL);
   
   // Pass PVD to SDK during initialization
   PxPhysics* physics = PxCreatePhysics(PX_PHYSICS_VERSION, foundation, 
                                        PxTolerancesScale(), true, pvd);
   ```

2. **Stream data to PVD**:
   The PhysX SDK automatically streams data to PVD for scene visualization. For custom data:
   ```cpp
   // Get the PVD client
   PxPvdClient* client = physics->getPvdClient();
   if(client && client->isConnected())
   {
       // Get data stream
       PxPvdDataStream* dataStream = client->getDataStream();
       
       // Create instances and update properties
       dataStream->createInstance(myObject);
       dataStream->setPropertyValue(myObject, "propertyName", propertyValue);
   }
   ```

3. **Disconnect and release**:
   ```cpp
   // Disconnect and release PVD when done
   pvd->disconnect();
   pvd->release();
   transport->release();
   ```
